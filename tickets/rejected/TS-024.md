# TS-024: Merge schedule, analyze, and split into a unified /ticket-system-schedule

## Objective
Combine `/ticket-system-schedule`, `/ticket-system-analyze`, and `/ticket-system-split` into a single intelligent `/ticket-system-schedule` command that takes one or more ticket IDs, evaluates them holistically (validity, relevance, atomicity, dependencies), proposes a plan (splits, rejections, ordering), and upon user approval moves everything to `planned/` or `rejected/`.

`/ticket-system-plan` is NOT affected — it keeps its role of activating a planned ticket into a worktree and generating implementation/test plans.

## Context
The current pipeline has three intermediate steps between backlog and the plan phase:
1. `schedule` — validates and moves a ticket from backlog to planned, inserts into roadmap
2. `analyze` — checks the first roadmap ticket for complexity and atomicity
3. `split` — if analyze says "needs split", proposes sub-tickets (which then need to be scheduled and analyzed again)

This creates unnecessary friction:
- The user must invoke up to 3 commands sequentially for a single conceptual action ("I want to work on these tickets")
- `analyze` always targets the first roadmap ticket, limiting flexibility
- `split` creates tickets in backlog, requiring another round of schedule+analyze — a loop
- The user already knows which tickets they want to prepare — the system should evaluate them in batch

The proposed unified `schedule` collapses this into one interaction with one approval gate. After schedule, tickets are ready for `/ticket-system-plan`.

## Acceptance Criteria
- [ ] `/ticket-system-schedule` accepts one or more ticket IDs as arguments (e.g., `/ticket-system-schedule TS-011 TS-012 TS-013`)
- [ ] The command also accepts a description to fuzzy-match tickets in backlog (existing behavior preserved for single ticket)
- [ ] For each ticket, the command evaluates:
  - **Validity**: frontmatter complete, acceptance criteria concrete and testable, technical approach sufficiently detailed. Refines if needed.
  - **Relevance**: is this ticket still needed given the current state of the codebase and completed tickets? Proposes rejection if obsolete.
  - **Atomicity**: 7-dimension complexity analysis (scope, criteria, cross-cutting, dependencies, risk, size, independence)
- [ ] For tickets that are too large (any High dimension, >3 criteria across concerns, >5 files), the command proposes a split with 2-4 sub-tickets including titles, scopes, and dependency chains
- [ ] For tickets that are no longer relevant, the command proposes rejection with reason
- [ ] The command resolves the full dependency graph across all input tickets and their proposed sub-tickets
- [ ] The command presents a unified scheduling plan: which tickets go to planned (with roadmap positions), which get split (with sub-ticket details), which get rejected (with reasons)
- [ ] **Human gate**: the user approves or adjusts the plan before any changes are made
- [ ] On approval: refined tickets `git mv` to `planned/`, sub-tickets are created directly in `planned/`, rejected tickets `git mv` to `rejected/`, roadmap.yml is updated with correct dependency+priority ordering
- [ ] All changes are committed: `TS-XXX, TS-YYY: Schedule tickets`
- [ ] `/ticket-system-analyze` is removed from the system (absorbed)
- [ ] `/ticket-system-split` is removed from the system (absorbed)
- [ ] The `ticket-system-reader` agent is removed if no other command uses it (help and next can use the editor agent or a remaining reader)
- [ ] `disable-model-invocation: false` (safe — requires user approval before any mutation)

## Technical Approach

### Pipeline simplification
```
BEFORE: create → schedule → analyze → (split → schedule → analyze) → plan → implement → verify → merge
AFTER:  create → schedule → plan → implement → verify → merge
```

Commands reduced from 9 to 7. The `plan` command is completely unchanged — it still activates a ticket from `planned/` into a worktree and generates implementation-plan.md + test-plan.md.

### What `/ticket-system-plan` continues to do (unchanged)
1. Verify `ongoing/` is empty and dependencies are completed
2. Create git worktree
3. Move ticket to `tickets/ongoing/PREFIX-XXX/` in worktree
4. Remove from roadmap
5. Generate implementation-plan.md and test-plan.md
6. STOP — human approval gate
7. User approves → `/ticket-system-implement`

### Agent selection
Use the editor agent (`$MID_MODEL`, `acceptEdits`). The analysis is structured (7-dimension checklist with defined thresholds) and split proposals follow templates. The editor already has all required bash permissions (git mv, git commit, git add, find, cat, mkdir, date).

If the analysis quality proves insufficient with `$MID_MODEL`, the fallback is to create a dedicated `ticket-system-scheduler` agent at `$STRONG_MODEL`. But try `$MID_MODEL` first.

### Behavior specification

**Phase 1 — Collect and resolve**
1. Read `.tickets/config.yml`
2. Parse arguments: extract ticket IDs or fuzzy-match description against backlog
3. For each ticket ID: read the ticket from `backlog/`, collect its dependencies
4. Recursively resolve dependencies: if a dependency is also in backlog, include it in the batch
5. Build a dependency graph of all tickets in scope

**Phase 2 — Evaluate each ticket**
For each ticket in the batch:
1. **Validate**: check frontmatter completeness, acceptance criteria quality, technical approach detail. If gaps exist, refine them (fill missing fields, sharpen criteria).
2. **Relevance check**: scan `completed/` and current codebase — is this ticket still needed? Has another ticket already solved the problem? Is the context obsolete?
3. **Atomicity analysis**: 7-dimension complexity assessment (same criteria as current analyze):
   - Scope (files/functions), Criteria (count, testability), Cross-cutting (layers), Dependencies (foundational work), Risk (unknowns), Estimated size (effort), Independence (testability)
   - Each rated Low / Medium / High
4. **Split decision**: if any dimension is High, or >3 criteria across concerns, or >5 files → propose split with 2-4 sub-tickets (titles, scopes, dependencies, rationale)

**Phase 3 — Present unified plan**
Present to the user:
```
SCHEDULING PLAN — 3 tickets evaluated

READY TO SCHEDULE:
  1. TS-011 "Migrate roadmap to YAML" — small, P0, no deps → roadmap position 1
  2. TS-013 "Add abort command" — small, P0, no deps → roadmap position 2

NEEDS SPLIT:
  3. TS-012 "Add doctor command" — medium but High on scope (7 checks)
     Proposed sub-tickets:
     a. TS-025 "Doctor: basic consistency checks" — small, deps: none
     b. TS-026 "Doctor: pending transaction detection" — small, deps: TS-025
     Accept split? [y/n/adjust]

PROPOSE REJECTION:
  (none)

Dependencies resolved: TS-011 before TS-013 (no actual dep, priority order)
```

**Phase 4 — Execute on approval**
1. Create sub-tickets directly in `planned/` (not backlog — they've already been evaluated)
2. `git mv` approved tickets from `backlog/` to `planned/`
3. `git mv` rejected tickets from `backlog/` to `rejected/`
4. Update `roadmap.yml` with all scheduled tickets in dependency+priority order
5. Update frontmatter: `status: planned` (or `rejected`), `updated: <now>`
6. Add log entries to each ticket
7. Commit all changes

### Commands removed
- `/ticket-system-analyze`: its 7-dimension analysis is now phase 2 of schedule
- `/ticket-system-split`: its split proposals are now phase 2 of schedule
