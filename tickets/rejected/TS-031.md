# TS-031: Remove --yes bypass from /ticket-system-schedule and replace with stop-on-conflict behavior

## Objective
Remove the `--yes` / `yes` bypass option from `/ticket-system-schedule` and eliminate the human-decision gate that currently blocks automated flows. Replace the gate with a deterministic behavior: if scheduling succeeds cleanly, proceed silently; if there is a conflict or ambiguity (ticket needs splitting, clarification required, dependencies unresolved), stop immediately and surface a clear message instructing the user to adjust the ticket via `/ticket-system-edit` before retrying.

## Context
The current `/ticket-system-schedule` command includes a human-decision gate (AskUserQuestion) and a `--yes` bypass intended to allow automated pipelines to skip the gate. In practice this is unreliable: the forked agent frequently misses the `--yes` flag, causing the pipeline to stall waiting for user input. This makes the command non-composable and frustrating to use from automated flows. The real need is simpler: scheduling either works or it does not. When it does not, the user should be told exactly what to fix and then manually adjust the ticket and re-run.

## Acceptance Criteria
- [ ] `/ticket-system-schedule` no longer accepts or acts on `--yes` / `yes` arguments.
- [ ] The human-decision AskUserQuestion gate is removed from the schedule skill.
- [ ] When scheduling succeeds (ticket is valid, no split needed, dependencies are clear), the ticket is moved to `planned/` and added to `roadmap.yml` without any user prompt.
- [ ] When scheduling detects a conflict or ambiguity (split needed, missing fields, dependency issues), the command stops, prints a clear explanation of the problem, and instructs the user to run `/ticket-system-edit TS-XXX` to resolve it before retrying `/ticket-system-schedule`.
- [ ] `specs.md` is updated to reflect the new schedule behavior (no gate, no bypass, stop-on-conflict model).
- [ ] `CLAUDE.md` is updated wherever it references the `--yes` bypass or human gate at the schedule stage.

## Technical Approach
- Locate the `/ticket-system-schedule` skill in `specs.md` section 4 (Command Pipeline).
- Remove all references to `--yes`, `yes` bypass, and AskUserQuestion from the schedule step.
- Define the conflict-detection heuristics inline: if the ticket has empty Objective, fewer than 2 acceptance criteria, unresolved dependency IDs, or the agent determines scope is too large (large complexity + vague criteria), that is a conflict.
- On conflict: emit a structured stop message listing the problems and the edit command to run.
- On clean pass: proceed directly to `git mv`, roadmap insertion, and commit.
- Update the "Human approval gates" bullet in `specs.md` section 6 (Key Design Decisions) and in `CLAUDE.md`.
- Update the validation checklist in `specs.md` section 8 to remove references to the schedule gate and `--yes` bypass.
