# TS-006: Add migration guide for updating installed ticket systems after spec changes

## Objective
Document a clear, step-by-step migration path for users who have an existing ticket system installation and want to upgrade it after a new version of `specs.md` is released.

## Context
As `specs.md` evolves, users with existing installs face an undocumented upgrade problem: they don't know which files changed, whether their `tickets/` data is compatible, or whether re-running `install.sh` is safe. Without guidance, users either stay on old versions (missing improvements) or blindly overwrite and risk breaking their setup.

**Audit review note (2026-04-04):** Deprioritized to P3 in alignment with TS-003 (versioning). Since the current usage model is regenerate-and-reinstall, a formal migration guide is premature. The simple migration path today is: regenerate from updated specs.md, re-run install.sh. Ticket data in `tickets/` is never affected by reinstallation (only agents and skills are overwritten). Revisit when versioning becomes a priority.

## Acceptance Criteria
- [ ] A `MIGRATION.md` file exists at the repository root
- [ ] `MIGRATION.md` has a section per version bump (starting from the first versioned release)
- [ ] Each section describes: what changed, which files to re-install, whether `tickets/` data needs any transformation
- [ ] `MIGRATION.md` is referenced from `README.md` (TS-001) in the installation section
- [ ] `install.sh` prints a reminder to check `MIGRATION.md` when upgrading an existing installation

## Technical Approach
- Create `MIGRATION.md` with a template section structure
- Add version detection logic to `install.sh`: if `$CLAUDE_DIR/ticket-system/VERSION` already exists, compare with the new version and warn the user to review `MIGRATION.md`
- Reference `MIGRATION.md` in `README.md`
