# TS-032: Remove --yes bypass from /ticket-system-plan and replace with stop-on-conflict behavior

## Objective
Remove the `--yes` / `yes` bypass option from `/ticket-system-plan` and eliminate the human-decision gate that currently blocks automated flows. Replace the gate with a deterministic behavior: if planning succeeds cleanly, proceed silently; if there is a conflict or ambiguity (plan is incomplete, acceptance criteria cannot be mapped to concrete steps, dependencies are unresolved), stop immediately and surface a clear message instructing the user to adjust the ticket via `/ticket-system-edit` before retrying.

## Context
The current `/ticket-system-plan` command includes a human-decision gate (AskUserQuestion) and a `--yes` bypass intended to allow automated pipelines to skip the gate. In practice this is unreliable: the forked agent frequently misses the `--yes` flag, causing the pipeline to stall waiting for user input. This makes the command non-composable and frustrating to use from automated flows such as `/ticket-system-run`. The real need is simpler: planning either works or it does not. When it does not, the user should be told exactly what to fix and then manually adjust the ticket and re-run.

## Acceptance Criteria
- [ ] `/ticket-system-plan` no longer accepts or acts on `--yes` / `yes` arguments.
- [ ] The human-decision AskUserQuestion gate is removed from the plan skill.
- [ ] When planning succeeds (ticket has clear objective, sufficient acceptance criteria, and all steps can be mapped), artifacts are written and committed without any user prompt.
- [ ] When planning detects a conflict or ambiguity (missing objective, fewer than 2 acceptance criteria, unmappable scope, unresolved dependency IDs), the command stops, prints a clear explanation of the problem, and instructs the user to run `/ticket-system-edit TS-XXX` to resolve it before retrying `/ticket-system-plan`.
- [ ] `specs.md` is updated to reflect the new plan behavior (no gate, no bypass, stop-on-conflict model).
- [ ] `CLAUDE.md` is updated wherever it references the `--yes` bypass or human gate at the plan stage.

## Technical Approach
- Locate the `/ticket-system-plan` skill in `specs.md` section 4 (Command Pipeline).
- Remove all references to `--yes`, `yes` bypass, and AskUserQuestion from the plan step.
- Define the conflict-detection heuristics inline: if the ticket has empty Objective, fewer than 2 acceptance criteria, unresolved dependency IDs, or the agent cannot decompose the work into concrete implementable steps, that is a conflict.
- On conflict: emit a structured stop message listing the problems and the edit command to run.
- On clean pass: proceed directly to writing plan artifacts and committing.
- Update the "Human approval gates" bullet in `specs.md` section 6 (Key Design Decisions) and in `CLAUDE.md`.
- Update the validation checklist in `specs.md` section 8 to remove references to the plan gate and `--yes` bypass.
