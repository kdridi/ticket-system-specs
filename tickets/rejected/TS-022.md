# TS-022: Add /ticket-system-run for automated pipeline orchestration

## Objective
Add a command that orchestrates the full pipeline from plan through merge, with human approval gates, automatic retries on verify failure, and circuit-breaking on repeated failures. This is the "ok/ok" automation — the user approves the plan once, then the system runs to completion.

## Context
The current workflow requires manual invocation of each step: plan → (approve) → implement → verify → (merge). The user's vision is to reduce this to: plan → (approve) → run → (merge). The run command would chain implement→verify automatically, retrying on FAIL up to MAX_RETRY times, and stopping for human intervention only when the circuit breaker trips or when merge is needed.

This is the most impactful strategic improvement, transforming the ticket system from a guided workflow into a semi-autonomous development pipeline.

## Acceptance Criteria
- [ ] A new skill `/ticket-system-run` exists with its own SKILL.md
- [ ] The command uses the `ticket-system-coder` agent (needs bypassPermissions for implementation + test execution)
- [ ] The command verifies prerequisites: ticket in ongoing/, plan exists and was approved (log entry present)
- [ ] The command executes the implement→verify cycle
- [ ] On VERDICT: PASS, the command reports success and suggests `/ticket-system-merge`
- [ ] On VERDICT: FAIL, the command automatically retries implement (passing the FAIL report as context) up to MAX_RETRY times
- [ ] On FAIL after MAX_RETRY attempts, the command stops and reports: "Plan may need revision. Review the failure report and consider re-planning."
- [ ] Each retry is logged in the ticket with the attempt number
- [ ] `disable-model-invocation: true` (autonomous execution — manual only)

## Technical Approach
Add to specs.md:
- New skill `ticket-system-run/SKILL.md` with `context: fork`, `agent: ticket-system-coder`
- The skill embeds the verification logic (does not fork again into the verifier — it runs tests and checks criteria itself)
- Behavior:
  1. Read `.tickets/config.yml`, read `test_command` if configured
  2. Verify prerequisites (ticket in ongoing, plan exists)
  3. Execute implementation plan (same logic as /ticket-system-implement)
  4. Run verification (same logic as /ticket-system-verify but inline)
  5. If PASS → move ticket to completed/, log, commit, suggest merge
  6. If FAIL → log attempt, read FAIL report, re-run implementation targeting only failed criteria
  7. Repeat 4-6 up to MAX_RETRY times
  8. If still FAIL → STOP, report, log "Circuit breaker tripped"

Key design decision: the run command embeds both implement and verify logic in a single agent session. This avoids the context loss of separate forks and allows the coder to see the verify FAIL report directly.

## Notes
- This is a large ticket that may benefit from splitting: (a) basic run with single attempt, (b) retry loop, (c) circuit breaker.
- The coder agent running verification inline means it self-assesses — this is less rigorous than an independent verifier. Trade-off: speed and context preservation vs independence.
- Future enhancement: option to use the verifier agent for independent verification between retries (at the cost of context loss).
- This ticket should be the last major feature before considering the system "v1.0 ready."
