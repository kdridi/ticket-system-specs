# TS-026: Absorb /ticket-system-split into /ticket-system-schedule

## Objective
Add split proposal generation and execution to `/ticket-system-schedule`. When the 7-dimension analysis (added in TS-025) flags a ticket as too large, schedule now proposes 2-4 sub-tickets with titles, scopes, and dependency chains. Remove `/ticket-system-split` as a standalone command.

## Context
TS-024 proposed merging schedule, analyze, and split into one command. That ticket was split into two phases. TS-025 (Phase A) merged schedule + analyze. This is Phase B: absorb split into the unified schedule command.

After TS-025, tickets flagged as "too large" are reported but not auto-split. This ticket closes that gap — schedule now handles the full lifecycle from evaluation through split proposals.

## Acceptance Criteria
- [ ] For tickets flagged as too large (any High dimension, >3 criteria across concerns, >5 files), `/ticket-system-schedule` proposes a split with 2-4 sub-tickets including titles, scopes, and dependency chains
- [ ] The unified scheduling plan now includes split proposals alongside ready/reject sections
- [ ] The user can accept, adjust, or reject split proposals at the human gate
- [ ] On approval: sub-tickets are created directly in `planned/` (not backlog — they've already been evaluated)
- [ ] The dependency graph resolution includes proposed sub-tickets (correct ordering)
- [ ] The original too-large ticket is updated to reference its sub-tickets
- [ ] `/ticket-system-split` is removed from the system (skill deleted, references removed from specs.md)
- [ ] The `ticket-system-reader` agent is removed if no other command uses it; otherwise retained
- [ ] Pipeline is fully simplified: `create → schedule → plan → implement → verify → merge`
- [ ] `disable-model-invocation: false` (safe — requires user approval before any mutation)

## Technical Approach

### Enhanced Phase 2 — Split proposals
Building on TS-025's evaluation phase, when a ticket is flagged as too large:
1. Propose 2-4 sub-tickets with:
   - Titles and scopes derived from the original ticket's acceptance criteria
   - Dependency chains between sub-tickets
   - Individual complexity estimates (should each be small or medium)
   - Rationale for the split boundaries
2. Sub-tickets inherit the parent ticket's priority and type

### Enhanced Phase 3 — Unified plan presentation
```
SCHEDULING PLAN — N tickets evaluated

READY TO SCHEDULE:
  1. TS-XXX "title" — small, P0, no deps → roadmap position 1

NEEDS SPLIT:
  2. TS-YYY "title" — medium but High on scope
     Proposed sub-tickets:
     a. TS-AAA "sub-title A" — small, deps: none
     b. TS-BBB "sub-title B" — small, deps: TS-AAA
     Accept split? [y/n/adjust]

PROPOSE REJECTION:
  (none)
```

### Enhanced Phase 4 — Execute splits
1. Create sub-tickets directly in `planned/` with sequential IDs
2. Update the original ticket to reference sub-tickets and mark appropriately
3. Include sub-tickets in roadmap.yml with correct dependency ordering
4. All other Phase 4 behavior from TS-025 remains unchanged

### Commands removed
- `/ticket-system-split`: its split proposals are now part of schedule's Phase 2
- Its skill (`ticket-system-split/SKILL.md`) is deleted

### Agent cleanup
- Evaluate whether `ticket-system-reader` is still needed by remaining commands (`/ticket-system-help`, `/ticket-system-next`)
- If no command uses it, remove the agent
- If commands still use it, retain and document

### Impact on other tickets
- **TS-016 (/ticket-system-next)**: update detection logic — no more "suggest analyze" or "suggest split" states; after backlog, suggest schedule
- **TS-023 (documentation)**: update pipeline description to final simplified form
