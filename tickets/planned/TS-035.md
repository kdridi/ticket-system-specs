# TS-035: Add timing/stats collection to /ticket-system-run

## Objective
Comprehensive observability for the ticket system pipeline. Two layers of instrumentation:
(A) Hook-based tool-level telemetry: PreToolUse and PostToolUse hooks that log every tool call (Read, Write, Edit, Bash, Grep, Glob, Agent, etc.) with timing, file paths, parameters, and session context. This runs at the Claude Code hook level and captures granular data across ALL phases automatically.
(B) Phase-level timing in /ticket-system-run: The orchestrator records wall-clock timestamps before/after each phase (plan, implement, verify, merge).
Both layers are opt-in via `stats: true` in `.tickets/config.yml`. Output is written to `.tickets/stats/` (gitignored). This gives empirical data to identify bottlenecks, detect redundant file reads across phases, and guide future optimization decisions.

## Context
Currently there is no way to know which phases take the longest, which tools are called most, which files get re-read across phases, or how run times vary by ticket complexity. The Claude Code hook system provides PreToolUse and PostToolUse events that fire on every tool call with full context (tool name, parameters, file paths, tool_use_id for correlation). By combining hook-level telemetry (per-tool-call granularity) with orchestrator-level phase timing, we get a complete picture of where time is spent. Key insight: correlating tool_use_id across PreToolUse and PostToolUse allows elapsed time calculation per tool call, and aggregating Read calls by file path across phases reveals redundant file reads — directly informing TS-036 (context artifact).

## Acceptance Criteria
- [ ] When `stats: true` is absent or false in `.tickets/config.yml`, no hooks are active, no stats files are written, behavior is unchanged.
- [ ] Two hook scripts are always installed with the system: `.claude/hooks/instrument-pre.sh` (PreToolUse) and `.claude/hooks/instrument-post.sh` (PostToolUse). These hooks are registered in `.claude/settings.json` at install time. At runtime, each hook reads `.tickets/config.yml` and exits 0 immediately if `stats: true` is not set — zero overhead when stats are disabled.
- [ ] The PreToolUse hook fires on every tool call and records to a temp state file: tool_use_id, tool_name, tool_input (file_path for Read/Write/Edit, command for Bash, pattern for Grep/Glob), start timestamp (epoch nanoseconds via `date +%s%N` or milliseconds via `date +%s%3N`), session_id, and cwd.
- [ ] The PostToolUse hook fires after every successful tool call, correlates with the PreToolUse entry via tool_use_id, computes elapsed_ms, and appends a complete log entry to `.tickets/stats/<session-or-ticket-id>.tool-calls.jsonl` (one JSON object per line).
- [ ] Each tool call log entry includes: timestamp, tool_name, tool_input summary (file path, command, or pattern — not full file contents), elapsed_ms, session_id, cwd, and tool_use_id.
- [ ] The PostToolUseFailure event is also hooked to capture failed tool calls with error info.
- [ ] `/ticket-system-run` records wall-clock timestamps before/after each phase and writes a phase-level summary to `.tickets/stats/<ticket-id>.json` with: ticket_id, title, type, priority, estimated_complexity, per-phase records (started_at, ended_at, duration_seconds), total_duration_seconds, run_status, and a tool_call_count per phase.
- [ ] Stats are written even when the run stops early due to failure.
- [ ] The `.tickets/stats/` directory is created if needed and is gitignored.
- [ ] Hook scripts are POSIX-compatible (bash, jq, date, standard utils only).
- [ ] Hook overhead is minimal — each hook script completes in under 50ms.

## Technical Approach
Two implementation layers:

**Layer 1 — Hook scripts:**
- Create `.claude/hooks/instrument-pre.sh`: reads JSON from stdin, extracts tool_use_id + tool_name + tool_input, records start_time to `.tickets/stats/.hook-state/<tool_use_id>.tmp` (one file per in-flight tool call).
- Create `.claude/hooks/instrument-post.sh`: reads JSON from stdin, looks up the matching `.hook-state/<tool_use_id>.tmp` file, computes elapsed time, appends a JSONL log line to `.tickets/stats/tool-calls.jsonl`, deletes the tmp file.
- Hook for PostToolUseFailure similarly, marking the entry as failed.
- Hook configuration goes in `.claude/settings.json` with matcher `".*"` to capture all tools.
- The hooks are conditional: they check for `stats: true` in `.tickets/config.yml` early and exit 0 immediately if stats are disabled (minimal overhead when off).

**Layer 2 — Orchestrator timing:**
- In `/ticket-system-run`, read `stats` flag from config at startup.
- Use `date +%s` before/after each Skill invocation.
- After run completion, write `.tickets/stats/<ticket-id>.json` with phase-level summary.
- Optionally parse the JSONL tool-call log to count tool calls per phase (by timestamp range).

**Output format — tool-calls.jsonl:**
```json
{"ts":"2026-04-08T14:23:45.123Z","tool":"Read","input":{"file_path":"src/auth.ts"},"elapsed_ms":45,"session":"abc123","tool_use_id":"toolu_01ABC"}
```

**Output format — <ticket-id>.json:**
```json
{"ticket_id":"PROJ-001","phases":[{"name":"plan","started_at":"...","ended_at":"...","duration_sec":45.2,"tool_calls":8}],"total_duration_sec":247.3,"status":"pass"}
```

## Notes
- `date +%s%N` gives nanosecond-precision epoch on Linux; on macOS it may not support %N — use `python3 -c "import time; print(int(time.time()*1000))"` or `gdate +%s%N` (coreutils) as fallback. The hook should detect platform and choose accordingly.
- The `.hook-state/` directory stores ephemeral per-tool-call temp files. These are cleaned up by PostToolUse. A SessionEnd hook or cleanup script should sweep stale files.
- JSONL format (one JSON object per line) is chosen over a single JSON array because it supports append-only writes and is jq-friendly (`jq -s '.' tool-calls.jsonl` to convert to array).
- The tool_input logged should be a summary (file path, command string, pattern) not the full content — avoid logging file contents or large outputs to keep log files small.
- Hook scripts must exit 0 on all paths to avoid blocking Claude's tool execution.
- Consider adding a `/ticket-system-stats` read-only command in the future to analyze collected data (not in scope for this ticket).
