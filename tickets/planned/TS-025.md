# TS-025: Merge /ticket-system-schedule and /ticket-system-analyze into unified schedule command

## Objective
Enhance `/ticket-system-schedule` to accept one or more ticket IDs, perform 7-dimension complexity analysis as part of scheduling, and present a unified plan before making changes. Remove `/ticket-system-analyze` as a standalone command (its logic is absorbed into schedule).

## Context
TS-024 proposed merging schedule, analyze, and split into one command. That ticket was too large and has been split into two phases. This is Phase A: merge schedule + analyze. Phase B (TS-026) will absorb split.

The current pipeline requires the user to invoke `schedule` then `analyze` sequentially. This ticket collapses them into a single interaction where the user provides ticket IDs and gets back a scheduling plan with complexity analysis included.

`/ticket-system-plan` is NOT affected — it keeps its role of activating a planned ticket into a worktree and generating implementation/test plans.

## Acceptance Criteria
- [ ] `/ticket-system-schedule` accepts one or more ticket IDs as arguments (e.g., `/ticket-system-schedule TS-011 TS-012 TS-013`)
- [ ] The command also accepts a description to fuzzy-match tickets in backlog (existing behavior preserved for single ticket)
- [ ] For each ticket, the command evaluates:
  - **Validity**: frontmatter complete, acceptance criteria concrete and testable, technical approach sufficiently detailed. Refines if needed.
  - **Relevance**: is this ticket still needed given the current state of the codebase and completed tickets? Proposes rejection if obsolete.
  - **Atomicity**: 7-dimension complexity analysis (scope, criteria, cross-cutting, dependencies, risk, size, independence)
- [ ] For tickets that are no longer relevant, the command proposes rejection with reason
- [ ] The command resolves the dependency graph across all input tickets
- [ ] The command presents a unified scheduling plan: which tickets go to planned (with roadmap positions), which get rejected (with reasons), which need further attention (too large — flagged but not auto-split in this phase)
- [ ] **Human gate**: the user approves or adjusts the plan before any changes are made
- [ ] On approval: refined tickets `git mv` to `planned/`, rejected tickets `git mv` to `rejected/`, roadmap.yml is updated with correct dependency+priority ordering
- [ ] All changes are committed: `TS-XXX, TS-YYY: Schedule tickets`
- [ ] `/ticket-system-analyze` is removed from the system (skill deleted, references removed from specs.md)
- [ ] `disable-model-invocation: false` (safe — requires user approval before any mutation)

## Technical Approach

### Pipeline simplification (Phase A)
```
BEFORE: create → schedule → analyze → (split → schedule → analyze) → plan → ...
AFTER:  create → schedule → plan → ...
```

Note: split is NOT yet absorbed in this phase. Tickets flagged as "too large" are reported to the user but not auto-split. The user can manually split or wait for TS-026.

### Agent selection
Use the editor agent (`$MID_MODEL`, `acceptEdits`). The analysis is structured (7-dimension checklist with defined thresholds). The editor already has all required bash permissions (git mv, git commit, git add, find, cat, mkdir, date).

### Behavior specification

**Phase 1 — Collect and resolve**
1. Read `.tickets/config.yml`
2. Parse arguments: extract ticket IDs or fuzzy-match description against backlog
3. For each ticket ID: read the ticket from `backlog/`, collect its dependencies
4. Recursively resolve dependencies: if a dependency is also in backlog, include it in the batch
5. Build a dependency graph of all tickets in scope

**Phase 2 — Evaluate each ticket**
For each ticket in the batch:
1. **Validate**: check frontmatter completeness, acceptance criteria quality, technical approach detail. If gaps exist, refine them.
2. **Relevance check**: scan `completed/` and current codebase — is this ticket still needed?
3. **Atomicity analysis**: 7-dimension complexity assessment:
   - Scope (files/functions), Criteria (count, testability), Cross-cutting (layers), Dependencies (foundational work), Risk (unknowns), Estimated size (effort), Independence (testability)
   - Each rated Low / Medium / High
4. **Flag decision**: if any dimension is High or complexity is large → flag as "needs attention" (user informed, but no auto-split in this phase)

**Phase 3 — Present unified plan**
Present to the user:
```
SCHEDULING PLAN — N tickets evaluated

READY TO SCHEDULE:
  1. TS-XXX "title" — complexity, priority, deps → roadmap position

NEEDS ATTENTION (too large):
  2. TS-YYY "title" — High on [dimensions]. Consider splitting before planning.

PROPOSE REJECTION:
  3. TS-ZZZ "title" — reason

Dependencies resolved: ordering rationale
```

**Phase 4 — Execute on approval**
1. `git mv` approved tickets from `backlog/` to `planned/`
2. `git mv` rejected tickets from `backlog/` to `rejected/`
3. Update `roadmap.yml` with all scheduled tickets in dependency+priority order
4. Update frontmatter: `status: planned` (or `rejected`), `updated: <now>`
5. Add log entries to each ticket
6. Commit all changes
