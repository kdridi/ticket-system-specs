# TS-003: Add versioning to generated system so installs can be audited

## Objective
Embed a version identifier into the generated output so that a user can always determine which version of `specs.md` was used to generate their installed ticket system, and whether they are running an outdated installation.

## Context
Currently, once `install.sh` copies files to `$CLAUDE_DIR`, there is no record of which spec version produced those files. If `specs.md` is updated and re-generated, users have no way to know their install is stale. Over time this creates silent divergence between the spec and the running system.

**Audit review note (2026-04-04):** This ticket was flagged as surengineering during the audit review. The current usage pattern is: user generates the system from specs.md and uses it as-is. Regeneration replaces everything. Versioning adds overhead without clear benefit in this mono-user, regenerate-on-demand model. Deprioritized to P3 — revisit only if multi-user distribution becomes a goal.

## Acceptance Criteria
- [ ] `specs.md` contains a `version:` field (e.g., `version: 0.1.0`) in a defined location (section 1 or a preamble)
- [ ] The generated output includes a `VERSION` file at its root containing this version string
- [ ] `install.sh` copies `VERSION` to `$CLAUDE_DIR/ticket-system/VERSION`
- [ ] `install.sh` prints the installed version at the end of a successful install
- [ ] The version string follows semver (MAJOR.MINOR.PATCH)

## Technical Approach
- Add a `version:` line near the top of `specs.md`
- Add a generation rule in section 5 instructing Claude Code to emit a `VERSION` file
- Update `install.sh` to copy and display the version
- Update the smoke test in section 8 to check for the `VERSION` file

## Notes
- Versioning does not need to be automatic (no git tags required at this stage). Manual bumps on meaningful spec changes are sufficient.
- A future ticket could add a `/ticket-system-version` slash command to show the installed version at runtime.
