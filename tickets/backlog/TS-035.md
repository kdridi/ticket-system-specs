# TS-035: Add timing/stats collection to /ticket-system-run

## Objective
When `stats: true` is set in `.tickets/config.yml`, the `/ticket-system-run` orchestrator records wall-clock timestamps before and after each phase (plan, implement, verify, merge) and writes a per-ticket JSON stats file to `.tickets/stats/<ticket-id>.json` with phase durations, total duration, and ticket metadata. This gives real measurement data to guide future optimization decisions.

## Context
Currently `/ticket-system-run` chains plan → implement → verify → merge with no timing instrumentation. There is no way to know which phases take the longest or how overall run times vary by ticket complexity. Adding opt-in stats collection (guarded by `stats: true` in config) provides empirical data for future optimization work without affecting projects that do not opt in.

## Acceptance Criteria
- [ ] When `stats: true` is absent or false in `.tickets/config.yml`, no stats file is written and behavior is unchanged.
- [ ] When `stats: true` is set, `/ticket-system-run` records a wall-clock start timestamp before each phase and an end timestamp after each phase completes (or fails).
- [ ] A JSON stats file is written to `.tickets/stats/<ticket-id>.json` upon completion (or failure) of the run.
- [ ] The stats file includes: `ticket_id`, `title`, `type`, `priority`, `estimated_complexity`, phase records (`plan`, `implement`, `verify`, `merge`) each with `started_at`, `ended_at`, and `duration_seconds`, plus `total_duration_seconds` and `run_status` (`pass` or `fail`).
- [ ] The `.tickets/stats/` directory is created if it does not exist.
- [ ] Stats are written even when the run stops early due to a phase failure (phases not reached are omitted from the output).
- [ ] No changes are made to the ticket file format or existing skill behavior outside of `/ticket-system-run`.

## Technical Approach
- Read `stats` flag from `.tickets/config.yml` at the start of `/ticket-system-run`.
- Use `date '+%Y-%m-%d %H:%M:%S'` (and `date +%s` for epoch seconds) to capture timestamps before and after each phase invocation.
- Accumulate phase timing data in-session.
- After the run concludes (all phases done or stop-on-failure triggered), construct the JSON object and write it to `.tickets/stats/<ticket-id>.json` using the `Write` tool.
- Use `mkdir -p .tickets/stats` via `Bash` before writing the file.
- The stats file is not committed — `.tickets/stats/` should be gitignored.
